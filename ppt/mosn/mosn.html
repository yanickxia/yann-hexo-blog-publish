服务治理与Mosn

author: yanick

服务治理的四个阶段

  • 单体 ：一般起始阶段业务很简单，流量也不大，所有的处理都可以在一个服务中完成；

  • 分布式 ：随着业务操作的多样化以及流量的日益增长，不得不按照服务维度进行拆分，这样相同的服务资源消耗可对等，方便容量评估及管理；

  • 微服务 ：随着服务的拆分粒度越来越细，其服务的数量一直在增加，由此出现各种微服务治理的需求（限流、鉴权、路由等），于是便出现各种治理组件并以 SDK 插件的方式集成到不同应用中；

  • Service Mesh ：伴随着服务治理的 SDK 种类、版本、重复等一系列问题，于是把 SDK 的能力剥离到 Sidecar，和业务进行解耦，从而实现业务和中间件能力的并行迭代；

服务治理的四个阶段 - 单体

优点：足够简单

服务治理的四个阶段 - 分布式架构

优点：架构上系统更加清晰，核心模块稳定，以服务组件为单位进行升级，避免了频繁发布带来的风险
开发管理方便

What’s MOSN

MOSN 是一款使用 Go 语言开发的网络代理软件，MOSN 是 Modular Open Smart Network 的简称。 MOSN 可以与任何支持 xDS API 的 Service Mesh 集成，亦可以作为独立的四、七层负载均衡，API Gateway、云原生 Ingress 等使用。

Why MOSN

  • 经过蚂蚁集团几十万容器的生产级验证
  • 基于 Go 语言易于开发
  • 与 Istio 完全集成

核心能力

  • 核心转发
    • 自包含的网络服务器
    • 支持 TCP 代理
    • 支持 TProxy 模式
  • 多协议
    • 支持 HTTP/1.1，HTTP/2
    • 支持 SOFARPC
    • 支持 Dubbo 协议
    • 支持 Tars 协议

核心能力

  • 核心路由
    • 支持 Virtual Host 路由
    • 支持 Headers/URL/Prefix 路由
    • 支持基于 Host Metadata 的 Subset 路由
    • 支持重试
  • 后端管理&负载均衡
    • 支持连接池
    • 支持熔断
    • 支持后端主动健康检查
    • 支持 Random/RR 等负载策略
    • 支持基于 Host Metadata 的 Subset 负载策略

核心能力

  • 可观察性
    • 观察网络数据
    • 观察协议数据
  • TLS
    • 支持 HTTP/1.1 on TLS
    • 支持 HTTP/2.0 on TLS
    • 支持 SOFARPC on TLS
  • 进程管理
    • 支持平滑 reload
    • 支持平滑升级
  • 扩展能力
    • 支持自定义私有协议
    • 支持在 TCP IO 层，协议层面加入自定义扩展

MOSN 核心概念

  • xDS（UDPA）支持
  • 业务支持
  • IO 模型
    • Golang 经典 netpoll 模型
    • RawEpoll 模型
  • 能力扩展
  • TLS 安全链路

MOSN 协程模型

  • 一条 TCP 连接对应一个 Read 协程，执行收包、协议解析；
  • 一个请求对应一个 worker 协程，执行业务处理，proxy 和 Write 逻辑；

常规模型一个 TCP 连接将有 Read/Write 两个协程，MOSN 取消了单独的 Write 协程，让 workerpool 工作协程代替，减少了调度延迟和内存占用。
